# Improvement Plan: Enhanced Form Validation & File Upload Support

## Goal
Improve the robustness of the dynamic Admin UI generated by Aperio by strictly enforcing OpenAPI schema validation rules (such as regex patterns, enums, min/max lengths) entirely on the client-side, and introduce comprehensive support for `multipart/form-data` file uploads.

## Problem
Currently, Aperio generates UI forms based on OpenAPI specs, but advanced validation rules defined in the specs (like strict regex validations, numeric bounds, and nested object rules) are sometimes bypassed or loosely enforced on the client side, resulting in unnecessary round-trips to the server only to receive a 400 Bad Request error. Additionally, APIs that require file uploads via `multipart/form-data` are notoriously tricky to handle in auto-generated UIs, and Aperio currently lacks robust support for drag-and-drop file inputs and binary payload management.

## Proposed Changes
1. **Ajv Integration Upgrade:** Leverage `@rjsf/validator-ajv8` alongside the React JSON Schema Form to enforce strict schema validation. Ensure that custom error messages are mapped nicely to the UI.
2. **Custom File Widget:** Create a custom Radix UI-based file upload widget (`FileWidget`) that supports drag-and-drop, multiple files, and size previews.
3. **Proxy Multipart Support:** Update the `src/app/api/proxy/route.ts` to detect when a payload is `multipart/form-data`. Ensure that the Node.js boundary streams are correctly handled and passed through to the destination API without corrupting binary data.
4. **Client-Side Form State Enhancement:** Add immediate visual feedback (e.g., green checkmarks, red text) for field-level validation as the user types, rather than waiting for form submission.

## Definition of Done
- Complex OpenAPI validation rules (regex, enums, min/max items, numeric boundaries) are strictly enforced in the UI before submission.
- User-friendly error messages appear below invalid fields immediately.
- A new file upload component allows users to attach files for `multipart/form-data` endpoints.
- The internal proxy successfully forwards file binary data to external target APIs.
- Comprehensive unit and integration tests are passing.
- Documentation is updated.

## Documentation Update
- Update `README.md` to note full support for OpenAPI validation keywords and `multipart/form-data`.
- Provide an example of how Aperio handles `format: binary` in OpenAPI specs.

## Testing
- **Unit Tests:** Test the proxy route handler's ability to parse and forward `multipart/form-data` payloads correctly.
- **Component Tests:** Verify that the Custom File Widget triggers the correct `onChange` events with valid File objects.
- **E2E Tests:** Use Playwright to load a spec with strict regex and max length rules, attempting to submit invalid data to ensure the submit button is blocked and errors are shown. Also simulate a file upload.

## Security
- **File Validation:** Implement a client-side file size limit and MIME-type check based on the OpenAPI spec constraints before allowing the file to be passed to the proxy.
- **Proxy Security:** Ensure the proxy does not cache or log binary file contents to prevent unintentional storage of sensitive user files.

## Reliability
- **Large Files:** Handle large file uploads gracefully in the proxy to avoid `Payload Too Large` (413) errors on the Next.js side by adjusting the bodyParser configuration for the proxy route.

## Accessibility
- Ensure the drag-and-drop file upload zone is fully operable via keyboard (e.g., using a visually hidden `<input type="file" />` that is focusable and triggerable via Space/Enter).
- Form validation error messages must have `aria-live="polite"` so screen readers announce them dynamically.
- Input fields with errors must have `aria-invalid="true"` and `aria-errormessage` linking to the error text ID.

## GDPR Compliance
- **File Upload Privacy:** Explicitly document that uploaded files are proxied directly to the user's configured endpoint and are not stored, scanned, or processed by Aperio in any way.
- **Data Minimization:** No metadata about the files or the validation errors is collected or sent anywhere other than the user's designated target API.
